# 批量生成前沿工业项目知识树的通用 Prompt（v6）

你将同时收到两份输入：
1. **项目 brief**：包含项目名称、来源母题、项目定位、完整项目逻辑、最终系统目标、粗略知识点、项目边界等信息。
2. **独立的 JSONC template 文件**：该文件只用于规定输出 JSON 的字段名称、层级结构与含义。**不要把 template 原样抄回输出**，而是按其字段要求填写内容，并最终输出**无注释的纯 JSON**。

你的任务不是生成一个“儿童玩具项目”，而是为一个**从约 10 岁儿童水平起步，但要经过数月到数年的持续学习，最终完成真实、完整、前沿、工业级项目**的学习者，构建一棵科学、可落地、可验收、可扩展的单项目知识树。

---

## 一、总目标
请把输入项目转化为一份**知识树 JSON**。这棵树必须同时满足以下目标：

1. **起点可进入**：学习者大约 10 岁，可以从较低门槛开始进入，但不是停留在儿童化玩具层面。
2. **终局工业级**：项目最终要指向一个真实的工业级系统、真实工作流、真实研发流程、真实物理系统或真实前沿科研平台。
3. **前沿且复杂**：项目必须属于当前科技前沿或近前沿领域，具有一定复杂度，不能是简单兴趣小制作。
4. **有趣且有画面感**：项目要有任务感、使命感、可展示成果，不能写成陌生行业黑话堆砌的晦涩项目。
5. **科学严谨**：所有节点、阶段、依赖关系、先后顺序、能力传承与验收标准，都必须符合科学性和工程逻辑。
6. **长期弥补认知差距**：知识树的作用是用完整规划逐步弥补儿童与工业级项目之间的知识差距，而不是跳过关键环节。

---

## 二、输出原则

### 1. 严格遵守 template
- **字段名称、层级结构、数组位置、对象嵌套**，都以 companion JSONC template 为准。
- 你可以根据项目需要**填充内容**，但不要随意删除模板中的关键结构。
- 最终输出必须是**合法纯 JSON**，不能带注释，不能带 markdown 代码块，不能带解释文字。

### 1.5 必须生成两个顶层“特殊节点”，并把它们当作知识树的一部分
- 在所有 stages 和 modules 之外，必须额外生成 **两个顶层特殊节点**：
  1. **总概述节点（project_overview）**：位于整棵树最前面；
  2. **未来扩展节点（future_extension）**：位于整棵树最后面。
- 这两个节点**不是普通教学模块**，也**不计入阶段模块数量**，但它们必须作为 JSON 结构中的正式节点存在。
- **总概述节点**必须清楚说明：
  - 这个项目到底是什么；
  - 它为什么真实、前沿、不是玩具；
  - 项目的背景、目标和完整任务逻辑；
  - 需要覆盖哪些知识领域；
  - 现实工业界有哪些真实落地项目、真实系统或真实团队在做类似事情；
  - 这个项目可用到哪些真实数据、公开数据、真实设备、真实工作流或真实平台。
- **未来扩展节点**必须清楚说明：
  - 如果完成这棵树，未来可以往哪些方向继续扩展；
  - 哪些方向更偏产品化、平台化、真机化、科研化或跨领域迁移；
  - 真实工业界还有哪些没有很好解决的问题；
  - 哪些是值得继续攻克的前沿难点或开放问题。
- 这两个特殊节点应服务于“看懂整棵树”和“看到未来路径”，不能写成空泛鸡汤，也不能写成教学管理说明。

### 2. 工业优先，不是儿童优先
- 儿童只是学习入口，不是项目定义标准。
- 项目描述应优先体现：真实工业对象、真实工程问题、真实研发流程、真实任务目标。
- 如果项目属于机器人、无人机、火箭、空间系统、边缘设备、实验自动化、能源系统、传感系统、制造系统等，终局原则上**不能停留在纯仿真**。
- 如果项目属于 bio-tech / health AI / 计算生物 / 医疗数据平台，终局可以是**真实辅助分析系统、真实科研工作流、真实实验自动化台架、真实证据链平台**，但不能写成危险湿实验或直接临床替代。

### 3. 节点数量 = f(概念密度)，绝不能平均分配
- **禁止所有里程碑/阶段使用相同的模块数量。**
- 节点数必须随着该阶段引入的**新概念数量**和**核心操作复杂度**而变化。
- 判断问题：这一阶段引入了多少个“学生之前不知道的新概念”？这些概念是否需要新的数据对象、工具链、算法思维、真实设备或系统接口？

推荐密度规则：
- **软性 / 认知 / 项目启动类阶段**：通常 **3–5 个模块**。
- **中等概念密度阶段**（标注、规则法、数据工程、接口整理、基线系统）：通常 **6–7 个模块**。
- **高概念密度阶段**（ML 建模、检测/分割、路径规划、证据融合、系统联调、真机闭环）：通常 **10–12 个模块**。

强制要求：
- **不要为了形式整齐而让每个阶段拥有相同数量的模块。**
- **不要为了显得完整而硬凑模块数量。**
- 如果某阶段只是铺垫，就少拆。
- 如果某阶段承担核心突破，就多拆。
- 输出中必须通过 `concept_density_class`、`new_concept_count_estimate`、`module_count_reason` 等字段体现为什么这个阶段节点多或少。

反例：
- 每个阶段固定 14 个节点，难度分布一致 —— 这是模板填充，不是教学设计。

### 4. 删除所有“教学管理”类节点
以下内容属于平台功能、教师行为或课程管理动作，**不是学生需要学习的项目知识**，不应占用模块：
- 个人能力评估、准备度雷达图
- 路径分流（A/B/C 入口选择）
- 学习合同、复评条件、打卡制度
- 泛化的“课程介绍”“学习约定”等与项目推进弱相关的节点

只有在这些内容直接决定真实项目边界、设备使用方式或数据合规条件时，才允许以**极小数量**保留。

### 5. 删除模板化填充节点
以下模式通常是为凑数而存在，**不得独立成模块**，除非它本身就是关键工程活动的一部分：
- “写出一张 XX 检查表”
- “做一次 XX 讲解/答辩/展示”
- “比较 X 和 Y 的区别”（如果 X 和 Y 已在前序模块中分别讲清）

这些内容可以作为某个模块内部的练习、讨论、附属交付物或验收动作，但不值得单独成为一个模块。

### 6. 每个阶段必须以“大作业 / 里程碑作品”收尾，而不是考试题
- 每个阶段都应以一个**capstone module** 结尾。
- 这个模块的产出必须是**有实质内容的大作业**，例如：
  - 一本词典
  - 一份标注数据集
  - 一个可运行脚本
  - 一张分类或分割图
  - 一个控制台原型
  - 一份报告
  - 一段台架演示视频
  - 一套接口与测试结果
- 验收形式不能只是选择题、问答题、口头测验。
- 大作业不是“最后顺手展示一下”，而是该阶段学习活动本身的重要组成部分。


### 6.5 每个阶段的大作业必须是“综合型工程实践”，而不是孤立作品
- 每个阶段最后一个 `capstone module` 必须是一个**阶段综合项目作业**。
- 这个大作业必须同时整合：
  1. **本阶段新获得的核心能力**；
  2. **前面阶段仍然有效且必须复用的关键产物**（如数据集、标签体系、规则库、脚本、接口、装置、台架、地图、控制逻辑、评测口径等）。
- 换言之，阶段大作业不是“学完这一阶段后单独做一道题”，而是要表现出：**学生已经能把前序成果接起来，做出一个更完整、更接近真实系统的小系统或子系统。**
- 大作业必须尽量体现**工程实践 / 动手实践**：
  - 对物理项目：应优先包含装配、接线、传感器接入、台架调试、机构安装、飞行/行驶/采集演练、现场测试、故障排查、重复验证等。
  - 对软件 / 数据 / AI 平台项目：应优先包含脚本编写、数据处理、标注、模型训练、评测、接口对接、前后端联调、日志检查、复现实验、部署验证等真实工程动作。
- **禁止**把阶段大作业写成纯汇报、纯答辩、纯讲解、纯测验、纯概念问答。
- 阶段大作业必须是“做一个东西并让它跑起来 / 用起来 / 测起来 / 验证起来”。
- 阶段说明中必须显式写明：
  - 该阶段大作业整合了哪些前序阶段产物；
  - 学生在这个大作业里要亲手完成哪些工程动作；
  - 这个大作业如何为下一个阶段提供可继续复用的资产。

### 7. 难度曲线必须前松后紧
整棵树的节点难度应符合真实学习曲线，而不是每阶段都均匀铺开：
- **前 1/3 阶段**：以低到中低难度为主，快速建立直觉、兴趣、任务感。
- **中间 1/3 阶段**：以中等难度为主，进入核心方法、核心工具、关键表示方式。
- **后 1/3 阶段**：以中高到高难度为主，进入工程实现、系统整合、真实验证、复杂误差处理。

强制要求：
- 不要让每个阶段内部都从最低难度到最高难度平均分布。
- 不要在最前面阶段塞过多高难度模块。
- 输出中要通过阶段说明、模块知识等级和模块顺序体现“前松后紧”。

### 8. 合并高度重叠的阶段或模块
如果两个阶段/模块的核心活动本质上是同一件事的两个连续子步，应当合并，而不是拆开凑数量。

判断标准：
- 是否共享同一批核心操作对象（同一批数据、同一套工具、同一个装置、同一种产出物）？
- 是否只是同一工作流中的相邻微步骤？
- 是否拆开后会导致教学意义下降、工程意义重复？

示例：
- “图像标注” + “风险标签设计” → 应合并
- “传统 ML 基线” + “CNN 训练” → 在很多项目里应合并为同一建模阶段下的连续模块，而不是两个远离的大阶段
- “项目边界” + “工程记录” → 若都属于启动阶段的软性内容，应合并

### 9. 主动补充被遗漏的桥梁概念
知识树不能只写显眼的大概念，还必须检查是否缺少连接两阶段的关键桥梁概念，尤其关注：
- **数据层面**：分辨率与识别的关系、数据泄漏、类别不平衡、训练/验证划分、样本偏差
- **模型层面**：过拟合的直觉、误差传播、阈值与召回/误报权衡、感知错误如何放大为决策错误
- **工程层面**：patch 到整图的拼接、模块接口格式、时间同步、日志口径统一、真实设备噪声

如果这些桥梁概念是阶段衔接所必需的，就必须进入树中；不能因为它们“不酷”就省略。

### 10. summary 必须是能力描述，不是概念介绍
每个模块的 `summary` 字段必须回答：**学完后学生能做什么**。
- 强制使用“**能够...**”句式。
- summary 应是**能力表述**，而不是课程介绍。

好例子：
- “能够根据症状文本和家族线索，把病例整理成可供候选基因排序的结构化表型清单。”
- “能够用颜色阈值和边缘强度两条规则，对一张 64x64 火星小图输出地貌类别。”

坏例子：
- “本节点聚焦规则法识别的核心概念，帮助学习者建立对阈值判断的基础认知。”

### 11. 必须体现“前后依赖”和“传承关系”
每个模块都必须说清楚：
- 它解决什么具体问题；
- 它继承了哪些前序能力、数据、规则、装置或接口；
- 它产出了什么可传递给后续模块的成果；
- 它为什么不能被跳过；
- 它与后续模块之间如何形成自然的工程传承；
- 哪些地方会形成反馈闭环（例如评测反馈、现场复盘、人工复核、误报修正、台架改版等）。

### 12. 每个模块都必须有可验收作品与标准
每个模块都必须带有：
- **acceptance_artifacts**：至少 1–2 项，最好是可演示、可测试、可交付的具体作品。
- **acceptance_standard**：至少 3 条，尽量具体、客观、可观察、可测试，不要只写“理解了”“学会了”。

验收作品可以是：
- 文档、图纸、原型、脚本、模型、报告、实验记录、台架、仪表板、控制台、演示视频、测试结果、数据集样例、接口定义、复盘清单等。


### 12.5 capstone module 必须显式写出“综合范围”和“动手证据”
对每个 `module_role = "capstone"` 的模块，必须额外保证：
- `capstone_scope` 体现它不是单阶段孤立作业，而是**整合当前阶段 + 前序相关阶段**；
- `integrates_previous_stage_outputs` 明确列出复用了哪些前序阶段产物；
- `hands_on_components` 列出学生亲手完成的工程动作；
- `engineering_practice_evidence` 说明什么证据能证明这不是纸面设计，而是真正做过、调过、测过、跑过；
- `acceptance_standard` 中至少有一条验证“综合整合”，至少有一条验证“动手实践”，至少有一条验证“结果可运行/可测试/可复现”。

### 13. 减少与主线弱相关的外围节点
- 不要堆太多“父母责任、安全教育、通用品德教育、泛泛而谈的科学精神”之类外围节点。
- 只保留对项目真正必要的边界、数据使用范围、真实设备限制、监督条件、合规要求。
- 主线应集中在：**输入 → 子系统 → 集成 → 测试 → 复盘 → 平台化 / 真机化 / 工业闭环**。

### 14. 标题和描述既要酷，也要真实
- 项目、阶段、模块标题可以更有任务感、使命感、画面感。
- 但不能为了“酷”而牺牲工业真实性。
- 如果项目 brief 本身用语过于行业黑话，可以在保持工业本体不变的前提下，把阶段或模块标题改得更有吸引力。

## 三、建议生成流程（内部遵循，不要输出）
1. 先判断项目的**工业本体**到底是什么。
2. 再判断项目的**真实终局**是什么：真机、工作流、平台、实验台、数字系统、数据闭环还是混合形态。
3. 再判断它对 10 岁起步学习者的**入口方式**是什么。
4. 先生成两个顶层特殊节点：
   - 最前面的**总概述节点**；
   - 最后面的**未来扩展节点**。
5. 再评估复杂度，并决定：
   - 需要多少阶段；
   - 哪些阶段节点少；
   - 哪些阶段节点多；
   - 哪些模块必须细拆；
   - 哪些模块应合并，不要凑数。
6. 检查是否遗漏桥梁概念。
7. 检查每个阶段最后是否都有**capstone module**，并确认该模块整合了本阶段与前序阶段的关键产物。
8. 检查每个阶段的大作业是否包含真实工程实践 / 动手实践，而不是展示或测验。
9. 检查所有模块 `summary` 是否都写成“能够...”能力句。
10. 检查总概述节点是否足以让读者在不看后文的情况下理解项目背景、目标、现实落地和数据来源。
11. 检查未来扩展节点是否给出了可信的延伸方向、产业落地方向与未解决问题。
12. 按 template 结构输出 JSON。

## 四、特别注意
- 输出必须是**纯 JSON**，不要带注释。
- 不要在输出前解释你的思路。
- 不要把模板本身再复述一遍。
- 不要用完全对称的节点数量。
- 不要把项目降级成玩具。
- 不要跳过关键依赖关系。
- 不要生成空泛、不可验收的节点。
- 不要把项目写成无聊、晦涩、只有行业黑话的结构。
- 不要生成以考试题或选择题为主的验收节点。
- 不要为了凑数保留教学管理节点或模板化填充节点。

- 不要遗漏最前面的总概述节点和最后的未来扩展节点。
- 不要把这两个特殊节点混进普通 stages/modules 里。
